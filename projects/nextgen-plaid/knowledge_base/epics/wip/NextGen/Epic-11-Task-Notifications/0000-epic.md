**Epic 11: Task Management & Notification System with BlueBubbles iMessage Integration**

**Epic Overview**
Build a unified task management and notification system that combines Kanban-style task tracking with multi-channel notification delivery (in-app toasts + iMessage via BlueBubbles). Tasks are modeled as artifacts (type: task) with status columns (To Do, Doing, Done), displayed on a drag-and-drop Kanban board, and creatable via natural language commands in the AI advisor chat. Notifications are delivered in-app via Turbo Stream DaisyUI toasts and optionally pushed to iMessage via the locally-hosted BlueBubbles REST API — eliminating the need for Twilio or any third-party messaging service. This epic merges the original backlog Epics 7 (Notification System) and 11 (Task Management) into a single cohesive epic, since task assignment and notification delivery are tightly coupled.

**User Capabilities**
- Interns see a Kanban board at `/tasks` with To Do, Doing, and Done columns — drag cards to update status
- Advisor says "Assign Monte Carlo simulation on portfolio" → task card appears on the board automatically
- Interns receive in-app toast notifications on task assignment, status changes, sync events, and advisor alerts
- Opted-in users receive iMessage notifications via BlueBubbles for high-priority events (task assigned, sync failed, allocation drift >10%)
- Users configure notification preferences: in-app only, in-app + iMessage, quiet hours
- Task cards show title, description, assignee, due date, priority, and status with real-time updates via Turbo Streams
- Advisor can create, update, and close tasks via natural language in chat (Epic 4 integration)
- Progress auto-updates from background jobs (e.g., sync complete → task marked done)

**Fit into Big Picture**
Structures the educational curriculum as actionable, trackable tasks — turning AI advisor suggestions into homework that interns can manage and complete. The notification layer keeps interns engaged between sessions with timely nudges (sync alerts, task reminders, advisor messages) delivered through their native messaging app (iMessage), matching how 22-30 year olds actually communicate. BlueBubbles runs locally on the same Mac infrastructure, maintaining the project's privacy-first architecture — no data leaves the local network for notifications. This epic enables future curriculum delivery (advisor assigns learning modules as tasks), proactive financial alerts (allocation drift, large deposits), and parent/admin visibility into intern progress.

### Key Decisions Locked In

**Architecture / Boundaries**
- **Tasks as Artifacts**: Extend the existing `Artifact` model with `artifact_type: "task"` rather than creating a separate Task model. Leverage existing JSONB `payload` for task-specific fields (description, due_date, priority, assignee_id, comments). Leverage existing `phase` for status mapping (repurpose or add task-specific statuses).
- **Notification model**: NEW `Notification` model (separate from Artifact) — stores channel, recipient, event_type, payload, read/delivered status, and timestamps. Polymorphic `notifiable` association to link notifications to their source (task, sync event, etc.).
- **BlueBubbles client**: NEW `BlueBubbles::Client` service object wrapping the local REST API (`http://localhost:1234/api/v1/`). Authentication via `guid` query parameter (server password stored in Rails credentials). No Twilio dependency.
- **BlueBubbles webhooks**: Rails endpoint receives inbound iMessage events (new messages, delivery receipts) via HTTP POST from BlueBubbles server. Enables future inbound text → advisor chat routing.
- **User preferences**: Add `notification_preferences` JSONB column to `users` table — stores channel toggles (in_app, imessage), quiet hours (start/end). Add `phone_number` string column directly on `users` table (NOT inside JSONB) — this is a first-class field for iMessage delivery via BlueBubbles and future SMS/voice features. Phone number should be validated (E.164 format) and encrypted at rest via `attr_encrypted` consistent with existing patterns.
- **Scope boundary**: This epic covers task CRUD + Kanban UI + notification delivery (in-app + iMessage). It does NOT include: inbound iMessage → advisor chat routing (future epic), rich media/MMS in notifications (future), or parent/admin task dashboards (future).

**UX / UI**
- Kanban board at `/tasks` — three columns: To Do, Doing, Done
- Drag-and-drop via Stimulus controller (SortableJS or native HTML5 drag API)
- Task cards: DaisyUI card component with title, assignee avatar/initials, priority badge, due date
- Toast notifications: DaisyUI toast component, bottom-right, auto-dismiss after 5s, stackable (max 3 visible)
- Notification center: Bell icon in navbar with unread count badge; dropdown shows recent notifications
- Mobile: Kanban columns stack vertically; swipe or tap to change status; toast notifications remain bottom-right
- **Selectable themes**: User-selectable DaisyUI theme (stored in `notification_preferences` JSONB or a dedicated `theme` column on users). Theme selector in navbar or settings page. Persist selection across sessions. Ship with curated set of DaisyUI themes appropriate for professional finance app (e.g., business, corporate, dark, night, luxury). Default: `business` (current app default).

**Testing**
- Minitest for all tests (per `.junie/guidelines.md`)
- ViewComponent tests for Kanban board, task card, toast, notification center
- Integration tests for BlueBubbles client (VCR cassettes for API calls)
- System tests (Capybara) for drag-and-drop task management and notification display
- Stimulus controller tests for drag-drop and toast behavior

**Observability**
- `Rails.logger.info` for: task creation/update, notification sent (channel + recipient), BlueBubbles API call (endpoint + status), webhook received
- `Rails.logger.error` for: BlueBubbles API failures, webhook parsing errors, notification delivery failures
- No Sentry yet — keep Rails.logger (consistent with Epic 4 decision)

### Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| BlueBubbles server offline/unreachable | iMessage notifications silently fail | Graceful fallback: log error, mark notification as `failed`, retry via background job (max 3 attempts). In-app toast always works as baseline. |
| Artifact model overloaded with task semantics | Schema confusion between artifact phases and task statuses | Clear separation: task-specific statuses stored in `payload["task_status"]`, not in the `phase` column which remains for SDLC workflow. OR: add a `status` column specifically for tasks. Architect to decide. |
| Drag-and-drop accessibility | Keyboard-only users can't reorder | Provide keyboard-accessible status change via dropdown on each card (WCAG 2.1 AA). Drag-and-drop is progressive enhancement. |
| BlueBubbles API changes | Client breaks silently | Pin to known API version (v1). VCR cassettes in tests catch contract changes. Health check endpoint in client. |
| Notification spam | Users overwhelmed by toasts/iMessages | Rate limiting: max 1 iMessage per event type per 5 minutes. Quiet hours enforcement. User can disable iMessage entirely. Toast auto-dismiss + max 3 visible. |
| Privacy — phone number stored in DB | PII exposure | `phone_number` stored as `attr_encrypted` column on users table (same pattern as Plaid access tokens). Per-record random IV. Never log raw phone numbers. |

### Key Guidance for All PRDs in This Epic

- **Architecture**: Tasks are artifacts (`artifact_type: "task"`), NOT a separate model. Notifications are a separate `Notification` model with polymorphic `notifiable`. BlueBubbles client is a standalone service in `app/services/blue_bubbles/`. All notification delivery goes through `NotificationDispatcher` — never send directly from controllers or jobs.
- **Components**: ViewComponents under `app/components/tasks/` (KanbanBoardComponent, TaskCardComponent) and `app/components/notifications/` (ToastComponent, NotificationCenterComponent, NotificationItemComponent). Follow DaisyUI + Tailwind patterns from Epic 3/4/5.
- **Data Access**: Controller scopes tasks to `current_user` family. Use `Artifact.where(artifact_type: "task")` with additional scopes. Defensive coding: handle missing assignee, nil due_date, empty payload gracefully.
- **Error Handling**: BlueBubbles failures are non-fatal — always log, never raise to the user. In-app toast is the reliable baseline; iMessage is best-effort. DaisyUI toast for user-facing errors (task save failed, etc.). Flash alerts for critical failures only.
- **Empty States**: Kanban board empty: "No tasks yet — your advisor will assign them as you progress through the curriculum." Notification center empty: "All caught up! No notifications."
- **Accessibility**: WCAG 2.1 AA. Drag-and-drop is progressive enhancement — always provide keyboard-accessible dropdown for status change. Bell icon has `aria-label="Notifications"` with count. Toast has `role="alert" aria-live="polite"`. Kanban columns have `role="list"` with `aria-label`.
- **Mobile**: Kanban columns stack vertically on mobile. Task cards are full-width. Toast stays bottom-right. Notification center becomes a full-screen drawer on mobile. Touch targets ≥44×44px.
- **Security**: Tasks scoped to user's family. BlueBubbles server password in Rails credentials (`Rails.application.credentials.dig(:blue_bubbles, :password)`). Phone number stored as encrypted string on users table via `attr_encrypted` (consistent with Plaid access token pattern). Webhook endpoint validates request origin (localhost or configured BlueBubbles host).
- **BlueBubbles API**: Base URL configurable via `Rails.application.credentials.dig(:blue_bubbles, :host)` defaulting to `http://localhost:1234`. Auth via `guid` query param. All calls wrapped in timeout (5s) and rescue. Health check: `GET /api/v1/ping?guid=<password>`.
- **Turbo Streams**: Task status changes broadcast to `tasks:#{current_user.family_id}` stream. Notifications broadcast to `notifications:#{current_user.id}` stream. Toast broadcast to `toasts:#{current_user.id}` stream.
- **Observability**: `Rails.logger.info "[Notification] dispatched channel=#{channel} user=#{user.id} event=#{event_type}"`. `Rails.logger.info "[BlueBubbles] sent to=#{handle} status=#{response.status}"`. `Rails.logger.error "[BlueBubbles] failed to=#{handle} error=#{error.message}"`.
