### PRD-1-05.5: Complete Plaid Field Mapping, Daily Balance Tracking & Denormalize Enrichments

**Overview**
Map all non-null Plaid fields that are currently being dropped; create unified daily balance snapshot model for accounts and liabilities; ensure proper credit card product name mapping; deprecate `enriched_transactions` table by migrating data back to `transactions`.

**Problem Statement**
Analysis of Plaid API responses reveals we're dropping critical data:
- **Accounts**: Missing `official_name` (credit card product names like "Prime Visa", "Ultimate Rewards®")
- **Accounts**: Missing `balances.limit` (credit card limits)
- **Transactions**: Missing `type` field (100% populated: buy/sell/fee/cash/transfer)
- **Transactions**: Missing `investment_transaction_id` and `security_id` references
- **Securities**: Missing pricing data (`close_price`, `close_price_as_of`) 86% populated
- **Securities**: Missing `ticker_symbol` (87% populated), `market_identifier_code` (83% populated)
- **No historical balance tracking**: Current balances overwritten on each sync, losing time-series data
- **Over-normalized enrichments**: `enriched_transactions` table requires JOINs; transactions are not normalized elsewhere; split provides no benefit and blocks Epic 2/3

### 2. Daily Balance Snapshots Model

**Create new model: `AccountBalanceSnapshot`**

**Purpose:**
- Track daily balance history for all accounts (investment, depository, credit, loan)
- Replace current pattern of overwriting `accounts.current_balance`
- Support liability tracking (credit card balances, loans)
- Enable time-series analysis for net worth dashboards

**Schema:**
```ruby
# Migration: create_account_balance_snapshots
create_table :account_balance_snapshots do |t|
  t.references :account, null: false, foreign_key: true
  t.date :snapshot_date, null: false

  # Balance fields (from Plaid balances object)
  t.decimal :available_balance, precision: 15, scale: 2
  t.decimal :current_balance, precision: 15, scale: 2
  t.decimal :limit, precision: 15, scale: 2  # credit cards
  t.string :iso_currency_code, default: "USD"

  # Liability-specific fields (for credit cards, loans)
  t.decimal :apr_percentage, precision: 5, scale: 2
  t.decimal :min_payment_amount, precision: 15, scale: 2
  t.date :next_payment_due_date
  t.boolean :is_overdue, default: false

  # Metadata
  t.datetime :synced_at
  t.string :source, default: "plaid"  # plaid, manual, calculated

  t.timestamps

  # Ensure one snapshot per account per date
  t.index [:account_id, :snapshot_date], unique: true, name: "index_balance_snapshots_unique"
  t.index :snapshot_date
  t.index [:account_id, :synced_at]
end
```

**Model:**
```ruby
class AccountBalanceSnapshot < ApplicationRecord
  belongs_to :account

  validates :snapshot_date, presence: true
  validates :account_id, uniqueness: { scope: :snapshot_date }

  scope :for_date, ->(date) { where(snapshot_date: date) }
  scope :for_date_range, ->(start_date, end_date) {
    where(snapshot_date: start_date..end_date).order(:snapshot_date)
  }
  scope :latest, -> { order(snapshot_date: :desc).limit(1) }
  scope :liabilities, -> { joins(:account).where(accounts: { type: ['credit', 'loan'] }) }

  # Derived fields
  def utilized_balance
    return nil unless limit && current_balance
    limit - available_balance
  end

  def utilization_rate
    return nil unless limit && limit > 0
    (utilized_balance / limit * 100).round(2)
  end
end
```

**Sync Strategy:**
```ruby
# In PlaidAccountSyncService or similar:
def sync_account_balance(account, plaid_balance_data)
  AccountBalanceSnapshot.create_or_find_by!(
    account: account,

    snapshot.current_balance = plaid_balance_data['current']
    snapshot.limit = plaid_balance_data['limit']
    snapshot.iso_currency_code = plaid_balance_data['iso_currency_code']

  end

  # Also update accounts table for quick access

    available_balance: plaid_balance_data['available'],
    current_balance: plaid_balance_data['current']
  )
end
```

### 4. Deprecate enriched_transactions Table (CRITICAL FOR EPIC 2/3)

**Problem:** Over-normalized for no benefit
- Transactions are not normalized elsewhere (we don't split out amounts, dates, etc.)
- Requires JOIN for every query that needs enrichment data
- Fields already exist in `transactions` table
- No other table references `enriched_transactions`
- **Blocks Epic 2/3**: Snapshot generation needs enrichment data without JOINs

**Solution:** Migrate data back to `transactions` table

```ruby
# Migration: migrate_enriched_transactions_back
class MigrateEnrichedTransactionsBack < ActiveRecord::Migration[8.0]
  def up
    # Migrate data from enriched_transactions back to transactions
    execute <<~SQL
      UPDATE transactions t
      SET
        logo_url = et.logo_url,
        website = et.website,
        merchant_entity_id = et.merchant_entity_id,
        merchant_name = et.merchant_name,
        personal_finance_category_id = et.personal_finance_category_id,
        personal_finance_category_icon_url = et.personal_finance_category_icon_url,
        personal_finance_category_confidence_level = et.personal_finance_category_confidence_level,
        personal_finance_category_version = et.personal_finance_category_version
      FROM enriched_transactions et
      WHERE t.id = et.transaction_id
        AND et.logo_url IS NOT NULL OR et.website IS NOT NULL
        OR et.merchant_entity_id IS NOT NULL OR et.merchant_name IS NOT NULL
        OR et.personal_finance_category_id IS NOT NULL
    SQL

    Rails.logger.info "Migrated enrichment data from enriched_transactions to transactions"
  end

  def down
    # Reverse migration (populate enriched_transactions from transactions)
    execute <<~SQL
      INSERT INTO enriched_transactions (
        transaction_id,
        logo_url,
        website,
        merchant_entity_id,
        merchant_name,
        personal_finance_category_id,
        personal_finance_category_icon_url,
        personal_finance_category_confidence_level,
        personal_finance_category_version,
        created_at,
        updated_at
      )
      SELECT
        id,
        logo_url,
        website,
        merchant_entity_id,
        merchant_name,
        personal_finance_category_id,
        personal_finance_category_icon_url,
        personal_finance_category_confidence_level,
        personal_finance_category_version,
        NOW(),
        NOW()
      FROM transactions
      WHERE logo_url IS NOT NULL OR website IS NOT NULL
        OR merchant_entity_id IS NOT NULL OR merchant_name IS NOT NULL
        OR personal_finance_category_id IS NOT NULL
      ON CONFLICT (transaction_id) DO UPDATE SET
        logo_url = EXCLUDED.logo_url,
        website = EXCLUDED.website,
        merchant_entity_id = EXCLUDED.merchant_entity_id,
        merchant_name = EXCLUDED.merchant_name,
        personal_finance_category_id = EXCLUDED.personal_finance_category_id,
        personal_finance_category_icon_url = EXCLUDED.personal_finance_category_icon_url,
        personal_finance_category_confidence_level = EXCLUDED.personal_finance_category_confidence_level,
        personal_finance_category_version = EXCLUDED.personal_finance_category_version
    SQL
  end
end
```

**Data Migration Considerations**

1. **Backfill Strategy for Snapshots:**
   - Option A: Start fresh from today (no backfill)
   - Option B: Create single snapshot per account from current `accounts` table values
   - Option C: Parse historical sync logs if available (complex)
   - **Recommendation**: Option A (start fresh) - simpler, cleaner

2. **Official Name Backfill:**
   - Fetch latest account details from Plaid on next sync
   - No historical data needed - current state is sufficient

3. **Security Price Data:**
   - Plaid provides current prices only
   - Historical prices require separate market data provider (out of scope)

**Out of Scope**
- Historical price data for securities (requires market data provider)
- Multi-currency support (USD only in V1)
- Intraday balance updates (daily snapshots only)
- Fixed income enrichment (Plaid returns null for all `fixed_income` fields)

**Known Limitations**
- **Fixed Income Fields**: Plaid API returns `fixed_income: null` for all securities (100% null rate). This is a Plaid data quality issue, not a mapping issue. See analysis in `tmp/plaid_transactions_parsed.txt`.
- **Transaction Fees**: Plaid's `fees` field is always 0 for this dataset. Fees may be embedded in `amount`.
- **Transaction DateTime**: Plaid's `transaction_datetime` is always null (only `date` is provided).
