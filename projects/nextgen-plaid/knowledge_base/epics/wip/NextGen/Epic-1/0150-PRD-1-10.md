Here is the third atomic **Product Requirements Document (PRD)** as requested, focused on enforcing a strict 1:1 relationship between each unique `security_id` and a single `SecurityEnrichment` record (FMP-only), while deprecating/removing Finnhub support (YAGNI principle—current repo shows both FMP and Finnhub in code, but we prioritize FMP for consistency and cost/simplicity).

**PRD 3: Enforce 1:1 Security (`security_id`) ↔ SecurityEnrichment (FMP-only), with Many Holdings per Security; Remove Finnhub Support**

**Overview**  
Make enrichment deterministic by ensuring each unique `security_id` has at most one active `SecurityEnrichment` record (FMP only). This yields a clean model where **many holdings can share the same `security_id`** (e.g., across accounts/lots), but they all resolve to the same enrichment record. Remove Finnhub integration to adhere to YAGNI, as FMP provides sufficient profile/quote/ratios/key_metrics for income projections, sector analysis, and AI prompts.

**Requirements**
- **Functional**:
    - Update `SecurityEnrichment` model association to reflect many holdings per security:
        - `has_many :holdings, primary_key: :security_id, foreign_key: :security_id, inverse_of: :security_enrichment`
    - Update `Holding` model association to resolve enrichment deterministically via `security_id`:
        - `has_one :security_enrichment, primary_key: :security_id, foreign_key: :security_id, inverse_of: :holdings`
    - Add uniqueness enforcement for the **security-level** 1:1:
        - App-level: `validates :security_id, uniqueness: true` (better errors)
        - DB-level: unique index on `security_enrichments.security_id` (see Non-functional)
    - In enrichment entrypoints:
        - Primary entrypoint: `app/jobs/sync_holdings_job.rb`
        - Enrichment logic: `app/services/security_enrichment_service.rb`
        - Refactor to **FMP-only** + **upsert by `security_id`**:
            - Use `find_or_initialize_by(security_id: holding.security_id)` (do not include `source` in the key)
            - Handle `ActiveRecord::RecordNotUnique` gracefully (reload existing record and retry/update)
    - Remove Finnhub references:
        - Remove `FINNHUB_API_KEY` from credentials / configuration.
        - Delete/comment Finnhub client code and call paths in `app/services/security_enrichment_service.rb`.
        - Remove Finnhub paths in `app/jobs/sync_holdings_job.rb` (or wherever enrichment is triggered).
        - Remove `finnhub` from any constants (e.g., `SecurityEnrichment::SOURCES`) and delete Finnhub-specific code branches.
        - Update any `Holding` delegated/derived methods that depended on Finnhub-only fields (e.g., `latest_consensus`, `target_upside_percent`) to be FMP-backed or `nil`.
        - Delete Finnhub VCR cassettes/fixtures under `spec/support/vcr_cassettes` (or update tests to FMP).
        - Search for and remove remaining `finnhub` references (grep/ripgrep) to prevent dead code.
    - Update accessor usage: In `Holding`, simplify `latest_enrichment_record` / `latest_enrichment_data` to delegate directly to `security_enrichment` (e.g., `def latest_enrichment; security_enrichment; end`).
    - Missing-field behavior after Finnhub removal:
        - Finnhub-only fields become `nil` (views should render a placeholder like "Not available" if needed).
- **Non-functional**:
    - Migration for index change: Add/ensure `add_index :security_enrichments, :security_id, unique: true`.
        - Note: `validates :security_id, uniqueness: true` is not DB enforcement by itself.
    - Zero-downtime cleanup + sequencing (explicit):
        1. Deploy code that stops creating Finnhub enrichments (remove Finnhub branch in `SecurityEnrichmentService`).
        2. Run a one-off cleanup rake task:
            - Delete all `SecurityEnrichment` rows where `source = 'finnhub'` (if `source` column still exists).
            - For any remaining duplicates per `security_id`: keep the record with max `enriched_at`, destroy others.
        3. Deploy migration adding the unique index on `security_id`.
        4. Later (separate PRD/migration): drop the `source` column once the system is stable and data is clean.
    - Privacy: No impact (enrichment data public).
    - Performance: deterministic lookup reduces query complexity (no `.order(enriched_at: :desc).first` needed).

**Architectural Context**
- Rails MVC: Associations in models only; no controllers/UI affected yet.
- PostgreSQL: Leverage existing `security_id` column (string, non-null); add unique index migration.
- Existing schema: `holdings.security_id` and `security_enrichments.security_id` already shared; no separate `Security` model exists.
- Jobs: Update enrichment trigger in `SyncHoldingsJob` + `SecurityEnrichmentService` to enforce security-level 1:1 upsert.
- AI: Simpler access for AiFinancialAdvisor prompts (e.g., `holding.security_enrichment&.data` directly).

**Acceptance Criteria**
- `Holding.first.security_enrichment` returns a single record (or nil); never an array/multiple.
- `SecurityEnrichment.first.holdings.count` reflects all holdings with matching `security_id`.
- Two holdings with the same `security_id` both resolve to the same single `security_enrichment`.
- Attempting to create a duplicate `SecurityEnrichment` for the same `security_id` is prevented by the DB unique index (and also raises/returns a validation error at the model layer).
- No Finnhub records exist post-cleanup (e.g., `SecurityEnrichment.where(source: 'finnhub').empty?` if the column still exists).
- `Holding#latest_consensus` (and similar) uses only FMP-backed data (or returns `nil`); no Finnhub logic remains.
- Rails console: `Holding.find(...).security_enrichment == SecurityEnrichment.find_by(security_id: holding.security_id)`.
- Enrichment job/service creates/updates only one record per `security_id` and gracefully handles `RecordNotUnique`.
- Tests pass; no Finnhub-related failures.
