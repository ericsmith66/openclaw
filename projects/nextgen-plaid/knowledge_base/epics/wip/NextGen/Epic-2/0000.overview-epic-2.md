# Epic 2: Build JSON Snapshots to Support Net Worth Dashboards

## Epic Overview
Implement daily/periodic FinancialSnapshotJob that exports structured JSON blobs capturing net worth, period-over-period changes, asset allocation breakdowns, and key aggregates from synced Plaid data. Adjusted for expanded v1 pages (net worth dashboard, asset allocation, sector weights, performance placeholder, holdings/transactions/liabilities summaries, curriculum/AI context). Epic 2 PRDs 2-03 through 2-05 assume SecurityEnrichment columns from Epic 1 PRD-0170 v2 are available with fallback to `holding.sector` or `data` JSON until fully deployed. Introduce Reporting::DataProvider service to centralize query logic, starting minimal for aggregates using ActiveRecord scopes and relations for composability. This promotes reusable patterns without overbuilding v1.

## User Capabilities
- Interns download/export personal snapshots
- Advisors (AI) ingest snapshots as RAG context for queries
- Admins preview/validate snapshot generation

## Fit into Big Picture
Turns raw syncs into portable, queryable financial context — essential for curriculum modules on net worth tracking, allocation drift, and historical performance without real-time computation overload. DataProvider enables evolution to grouped/shared accounts and BI exports.

## Timezone Policy
- **Storage**: All timestamps stored in UTC in database
- **Business Logic**: All date calculations use CST ('America/Chicago')
- **Display**: Present dates in CST to users
- **Implementation**: Use `APP_TIMEZONE` constant and `Date.current` (respects Rails time_zone config)
- **V1 Scope**: No per-user timezone support; all users see CST

## Target Snapshot JSON Structure v1

```json
{
  "schema_version": 1,
  "as_of": "2026-01-24",
  "total_net_worth": 2500000.00,
  "delta_day": 5000.00,
  "delta_30d": 15000.00,
  "asset_allocation": {
    "equity": 0.62,
    "fixed_income": 0.18,
    "cash": 0.15,
    "alternative": 0.05
  },
  "sector_weights": {
    "technology": 0.28,
    "healthcare": 0.15,
    "financials": 0.12,
    "unknown": 0.45
  },
  "top_holdings": [
    {"ticker": "AAPL", "name": "Apple Inc.", "value": 300000.00, "pct_portfolio": 0.12},
    {"ticker": "MSFT", "name": "Microsoft Corp.", "value": 250000.00, "pct_portfolio": 0.10}
  ],
  "monthly_transaction_summary": {
    "income": 10000.00,
    "expenses": 5000.00,
    "top_categories": [{"category": "Salary", "amount": 8000.00}, {"category": "Rent", "amount": -2000.00}]
  },
  "historical_net_worth": [
    {"date": "2026-01-23", "value": 2495000.00},
    {"date": "2025-12-25", "value": 2485000.00}
  ],
  "data_quality": {
    "score": 95,
    "warnings": ["Missing price for 2 holdings", "Stale sync detected"]
  },
  "disclaimer": "Educational simulation only – not financial advice"
}
```

### JSON Field Semantics
Consistent handling of empty/missing data:
- **Empty Arrays `[]`**: No items but feature is working (e.g., `"top_holdings": []` means no holdings)
- **Empty Objects `{}`**: No data available (e.g., `"asset_allocation": {}` means no positions)
- **Null `null`**: Feature not applicable (e.g., `"sector_weights": null` when user has no equities)
- **Specific Example**:
  ```json
  {
    "top_holdings": [],               // No holdings (valid state)
    "sector_weights": null,           // User has no equities
    "asset_allocation": {},           // No positions at all
    "monthly_transaction_summary": {
      "income": 0,
      "expenses": 0,
      "top_categories": []            // No transactions this month
    }
  }
  ```

Reference this structure in all subsequent PRDs for JSON extensions.

## Schema Versioning Policy

### Version Bump Triggers
- **Breaking changes only**:
  - New required field added
  - Existing field renamed
  - Field type changed
  - Field removed

### Reading Old Versions
- Use `Reporting::SnapshotAdapter.read(version, data)` to normalize to latest schema
- Adapter handles backward compatibility

### Migration Path
- Model method: `FinancialSnapshot.migrate_schema(old_version, data)` upgrades schema
- Full policy documented in `knowledge_base/data/snapshot-schema-policy.md`

## Status Enums
FinancialSnapshot status values:
- `:pending` - Snapshot job queued but not started
- `:complete` - Snapshot successfully generated
- `:error` - Snapshot generation failed (see data.error for details)
- `:stale` - Snapshot generated but underlying Plaid sync is >36h old
- `:empty` - Snapshot generated but user has no financial data
- `:rolled_back` - Snapshot invalidated by rollback operation
