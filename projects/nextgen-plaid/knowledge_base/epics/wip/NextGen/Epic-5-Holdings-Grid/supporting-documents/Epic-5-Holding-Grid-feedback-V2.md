## Questions

### PRD #1 (Data Provider Service)
1. **Snapshot JSON Structure**: Will snapshots include enrichment data inline, or will we join to `security_enrichments` table even in snapshot mode? This affects whether enrichment freshness is frozen at snapshot time or always shows current data.

2. **Multi-Account Grouping Key**: When aggregating multi-account holdings, are we grouping by `security_id` or `ticker_symbol`? Different securities can share tickers (e.g., CUSIP changes after merger), and same security might have missing ticker. Recommend grouping by `security_id`.

3. **Cache Key Strategy**: The 4h TTL cache mentions `filter_hash`. How is this hash computed? SHA256 of sorted params JSON? Need to ensure different filter combinations don't collide.

### PRD #2 (Saved Account Filters)
4. **Filter Criteria Schema**: The jsonb `criteria` field is flexible but undefined. Should we document the expected schema? Example:
   ```json
   {
     "account_ids": [1, 2, 3],
     "institution_ids": [10],
     "ownership_types": ["Trust"],
     "asset_strategy": "long_term",
     "trust_code": "TR001"
   }
   ```

5. **Cross-App Filter Application**: When used in net worth or transactions views, will the same SavedAccountFilter model work, or do those features need different filter fields? Should we make this extensible?

### PRD #3 (Core Table)
6. **Toast Warning Mechanism**: For the >400 rows warning, should this be dismissible? Should it remember dismissal per session? Consider user annoyance if they regularly work with large portfolios.

### PRD #4 (Filters & Tabs)
7. **Asset Class Mapping**: What are the exact Plaid `asset_class` values we're mapping to the 3 tabs? Need to ensure comprehensive coverage (crypto, options, alternatives, cash equivalents).

### PRD #5 (Search & Sort)
8. **Business Day Calculation**: For enrichment freshness, how do we handle market holidays (not just weekends)? Should we integrate a market calendar or use a simple heuristic (5 business days = 7 calendar days)?

9. **Search Performance**: Global search with ILIKE on multiple columns can be slow. Have we considered full-text search (tsvector/tsquery) or Postgres trigram indexes (pg_trgm)?

### PRD #6 (Multi-Account Expansion)
10. **Expand State Persistence**: How do we preserve expand/collapse state across pagination/filter changes? Storing in session? URL params? Stimulus controller with localStorage?

### PRD #7 (Security Detail Page)
11. **Transaction Grand Totals Logic**: What exactly goes into "total invested" vs "proceeds" vs "net cash flow"? Need clear definitions:
    - Invested = sum of buy amounts?
    - Proceeds = sum of sell amounts?
    - Net = invested - proceeds?
    - Dividends = sum of dividend transaction amounts?

12. **Security Not Found**: What happens if someone navigates to `/portfolio/securities/99999`? 404? Redirect with message?

### PRD #8 (Snapshots Model)
13. **Snapshot Naming**: The `name` field is optional. When auto-generated by scheduled job, what's the naming convention? "Daily Snapshot 2026-02-04"? Or just rely on created_at?

14. **Account-Level vs User-Level**: Can a single snapshot be mixed-scope (some accounts but not all)? Or is it binary: all accounts (account_id null) or single account (account_id present)?

### PRD #9 (Snapshot Creation)
15. **Idempotency Check**: The "<24h skip unless forced" logic — is this per user or per user+account combination? Could cause issues if user wants both user-level and per-account snapshots daily.

16. **Solid Queue Failure Handling**: If snapshot creation fails (Plaid down, DB issue), what's the retry strategy? Exponential backoff? Alert admin?

### PRD #10 (Snapshot Comparison)
17. **Security Matching Logic**: When comparing snapshots, do we match by `security_id`, `ticker_symbol`, or both? What happens if ticker changed between snapshots (corporate action)?

18. **Return Calculation Edge Cases**: How do we handle:
    - Division by zero (start value = 0)?
    - Negative cost basis (return of capital)?
    - Positions that moved accounts (show as removed + added)?

### PRD #11 (Snapshot Selector)
19. **Snapshot List Size**: If a user has 1000+ snapshots (multi-year daily), the dropdown will be unwieldy. Should we paginate, show only recent 50, or add a date picker?

### PRD #12 (Comparison Mode)
20. **Comparison Performance**: For 500 holdings compared, will this be computed on every page load or cached? Should we memoize comparison results with a cache key?

### PRD #14 (Export)
21. **Large Export Handling**: For portfolios with 1000+ holdings, CSV generation could timeout. Should we async this via Solid Queue and email the download link?

22. **Export Comparison Data**: When in comparison mode, does the CSV export include the delta columns? Or just the current view?

### PRD #15 (Mobile)
23. **Mobile Feature Parity**: Should mobile support all features (multi-account expansion, comparison mode, snapshots) or subset for v1? Complex interactions may not translate well to small screens.

## Suggestions

### Architecture & Performance
1. **Materialized View for Totals**: Consider a `portfolio_totals` materialized view refreshed nightly that pre-computes common aggregations. Would eliminate cache complexity and provide sub-50ms totals queries.

2. **Background Job Observability**: Add structured logging and metrics to all Solid Queue jobs (duration, success/failure counts, retry counts) for monitoring snapshot health.

3. **Security Enrichment Backfill**: Document the process for backfilling enrichment data when a new security appears. Does it happen on-demand at Holdings sync or via scheduled job?

4. **Database Indexes**: Explicitly list required indexes in each PRD's acceptance criteria. Examples:
   - `holdings(user_id, security_id, account_id)`
   - `security_enrichments(security_id, enriched_at)`
   - `holdings_snapshots(user_id, created_at DESC)`

### User Experience
5. **Keyboard Navigation**: Consider adding keyboard shortcuts (j/k for row navigation, x for expand/collapse, / for search) for power users.

6. **Loading States**: Define skeleton screens or spinners for data provider calls, especially snapshot loading and comparison mode.

7. **Tooltip Documentation**: Add DaisyUI tooltips for columns like "Cost Basis" explaining phase 1 limitations (sum only, not weighted average).

8. **Breadcrumb Navigation**: On Security Detail Page, show breadcrumb: Holdings Grid (with filters) > [Ticker] for clear navigation hierarchy.

9. **Empty State Variations**: Create distinct empty states for:
   - New user (no holdings synced yet)
   - Filtered to zero results
   - No snapshots created
   - Snapshot with zero holdings (all sold)

### Testing & Quality
10. **Performance Regression Tests**: Set up automated performance benchmarks in CI:
    - Data provider with 500 holdings < 500ms
    - Snapshot comparison < 2s
    - Grid render with "All" at 1000 holdings < 3s

11. **RLS Test Coverage**: Each model with RLS should have explicit test cases verifying:
    - User A cannot access User B's data
    - SQL injection attempts via `current_setting` are blocked
    - Superuser context works for admin tools

12. **Accessibility Testing**: Run automated a11y checks (axe-core) on each view as part of Capybara system tests.

13. **Edge Case Test Suite**: Create fixtures for pathological cases:
    - Security held in 10+ accounts
    - Portfolio with 2000 holdings
    - Snapshot with all removed positions
    - Negative unrealized G/L across entire portfolio

### Documentation
14. **Data Dictionary**: Create `knowledge_base/data-dictionary.md` documenting:
    - All jsonb schemas (snapshot_data, filter criteria)
    - Asset class enum values
    - Transaction type enum values
    - Enrichment data field definitions

15. **Runbook for Snapshots**: Document operational procedures:
    - How to trigger manual snapshot for all users
    - How to restore/delete corrupted snapshots
    - How to investigate missing/failed scheduled snapshots

16. **API-like Service Docs**: For data provider and comparator services, document inputs/outputs in RDoc or YARD format for future maintainers.

### Error Handling Strategy
Add to epic:

**Data Provider Errors**:
- Missing enrichment → Use stale data + show warning
- Snapshot not found → 404 with message "Snapshot no longer available"
- Invalid filter criteria → Default to "All Accounts" + toast warning
- Cache failure → Log error, compute fresh (graceful degradation)

**Snapshot Creation Errors**:
- Plaid timeout → Retry 3x exponential backoff, then log failure + alert
- Holdings query failure → Capture partial data if possible, flag as incomplete
- JSON serialization error → Log stack trace, skip snapshot
- Disk full → Alert immediately, block further snapshots

**Comparison Errors**:
- Missing snapshot → Show error message in UI
- Mismatched structure → Attempt best-effort comparison, show warning
- Division by zero → Treat as infinite return, display "N/A"

## Objections & Solutions

### Objection 1: RLS Policy Application Complexity
**Issue**: Setting `current_setting('app.current_user_id')` for every request adds overhead and complexity. Must be set in middleware or around every controller action. Easy to miss and create security holes.

**Solution**:
- Create `ApplicationController` before_action to set context:
  ```ruby
  before_action :set_rls_context

  def set_rls_context
    ActiveRecord::Base.connection.execute(
      "SET LOCAL app.current_user_id = #{current_user.id}"
    )
  end
  ```
- Add integration test that verifies RLS blocks cross-user access
- Consider using `apartment` gem for user-scoped schemas if RLS proves fragile
- Document RLS setup prominently in `knowledge_base/architecture/rls-setup.md`

### Objection 2: Snapshot JSON Queryability
**Issue**: Storing full holdings as jsonb limits query flexibility. Can't efficiently join snapshot holdings with enrichment table for freshness checks. Can't index nested fields without explicit GIN indexes.

**Solution**:
- For v1, accept limitation and parse JSON in Ruby
- For v2 (if needed), consider:
  - Normalized `snapshot_holdings` join table (snapshot_id, security_id, quantity, value, ...)
  - Or jsonb_path_query with GIN indexes on common access patterns
  - Or hybrid: keep jsonb for raw data + extract key fields to columns
- Monitor query performance and optimize only if slow

### Objection 3: Cache Invalidation Timing
**Issue**: "Nightly invalidation" is vague. If snapshots run at 1:30 AM but cache invalidates at 2:00 AM, users could see stale data. If holdings sync happens during the day (manual refresh), cache is stale until night.

**Solution**:
- Invalidate cache in same transaction as holdings update:
  ```ruby
  after_commit :invalidate_portfolio_cache, on: [:create, :update, :destroy]
  ```
- Set cache TTL to 1 hour (not 4) for more responsive updates
- Add manual "Refresh" button in UI that bypasses cache
- Document cache behavior: "Totals may be cached up to 1 hour for performance"

### Objection 4: Multi-Account Cost Basis Deferred
**Issue**: Showing summed cost basis without weighted average will produce incorrect "Unrealized G/L %" for multi-account securities. Users may rely on this for tax decisions, leading to errors.

**Solution**:
- For phase 1, **do not show** Unrealized G/L (%) column for multi-account aggregated rows — only show for expanded per-account rows where it's accurate
- Display tooltip on aggregated rows: "Expand for per-account cost basis and returns"
- Add disclaimer at top of page: "Multi-account summaries are approximate. Expand rows for precise cost basis."
- Prioritize weighted average cost basis as **PRD #16** (high value, moderate complexity)

### Objection 5: Snapshot Storage Growth Unmonitored
**Issue**: Even with <10 users, daily snapshots × 300 holdings × 365 days = ~1 million rows in JSON. Without monitoring, storage could silently fill disk.

**Solution**:
- Add database constraint + monitoring in PRD #8:
  ```sql
  CHECK (octet_length(snapshot_data::text) < 1048576) -- 1MB limit
  ```
- Add Prometheus/StatsD metric for snapshot table size
- Create alert when `pg_total_relation_size('holdings_snapshots') > 10GB`
- Implement retention policy in PRD #9: automatically delete snapshots >1 year old (except monthly)
- Add admin dashboard showing per-user snapshot count and size

### Objection 6: Export Synchronous for Large Datasets
**Issue**: PRD #14 uses synchronous CSV generation. For 1000+ holdings with multiple columns, this could exceed Heroku's 30-second timeout or lock browser.

**Solution**:
- Implement size check: if filtered count > 500, use async path
- Async path:
  ```ruby
  def export
    if @data_provider.total_count > 500
      ExportHoldingsJob.perform_later(current_user.id, filter_params)
      flash[:notice] = "Export started. You'll receive an email with download link."
    else
      # Synchronous CSV generation
    end
  end
  ```
- Store exports in S3/ActiveStorage with 24h expiration
- Send email with signed download URL
- Add exports index page showing recent exports + status
