# 0080-Ollama-Agentic-PRD-Spike-Epic

## Overview
This epic spikes an Ollama-based agentic workflow for PRD generation, using the existing multi-index RAG structure (functional/structural/vision + cross-references) to validate if it produces higher-quality outputs (structured, hallucination-resistant, aligned with vision) at zero marginal cost compared to cloud LLMs. Implementation as a standalone Ruby module for isolation; no codebase integration until validated. Human review required for all outputs before any code changes.

**Rationale**: Cloud LLM costs (e.g., Grok/Claude) are high; local Ollama (Llama3.1 70B/405B on M3 Ultra) enables offline experimentation. Leverage current RAG (sufficient for POC per rag-structure concat) without building/enhancing it here—defer RAG automation/insufficiency fixes to follow-on epic if validation succeeds.

**Scope**: 7-10 day spike; focus on 5-phase lifecycle (research → plan → verify → execute → validate) from Agenti-PRD-Generation.md, tested on 2-3 sample features (e.g., Plaid holdings sync extension). Goal: Confirm agentic flow + RAG yields better PRDs (e.g., 70%+ first-pass validation rate, 5-8 ACs, vision ties) vs. single-shot.

**Success Criteria**:
- Generate 2-3 sample PRDs with agentic flow; compare to manual/cloud versions on metrics: structure adherence (100% required sections), hallucination rate (<10%), AC quality (5-8 verifiable bullets), token efficiency (30%+ reduction via phased RAG).
- RAG supports structure: Phase-specific pulls (e.g., research: summaries only) without gaps; if insufficient (e.g., missing deltas), note for next epic but don't fix here.
- Ollama viability: <20s per phase on M3 Ultra; fallback to Claude 4.5 if reasoning weak.
- Byproduct: Basic RAG automation insights (e.g., delta summaries) documented, but not implemented.

**Out of Scope**: RAG enhancements (e.g., dynamic deltas, vector DB); full multi-agent (Coordinator/CWA); production integration; UI changes.

## Current Problems
- Single-shot PRD generation: High hallucinations, inconsistent structure (e.g., variable AC count), token waste from full RAG dumps.
- Cloud dependency: Expensive retries; privacy risks for financial data.
- RAG limitations: Static-heavy; no auto-deltas, but sufficient for spike validation—test if it grounds phases adequately.

## Proposed Solution
Standalone Ruby script (lib/agentic_prd_spike.rb) implementing 5-phase flow with Ollama calls via gbaptista/ollama-ai gem. Use existing RAG via mocked MultiIndexProvider (hardcoded from rag-structure indexes/config.yml). Human gates: Review phase outputs/PRDs before proceeding.

### High-Level Architecture
- **Phases**: Per prd-generation-questions.md, with JSON handoffs.
- **RAG Integration**: Query existing MultiIndexProvider for phase-specific context (e.g., research: 1K summaries from functional/vision; execute: 8K full from structural + on-demand schema).
- **Ollama**: Llama3.1:70b baseline; tune prompts for JSON outputs.
- **Validation**: Run on samples; manual human review + scoring.

### Implementation Plan
Break into atomic tasks; each as a potential PRD stub (detailed PRDs generated only if requested post-spike).

1. **Setup & Ollama Integration** (1-2 days):  
   Standalone Ruby env with ollama-ai gem; basic LLM call wrapper. Test simple prompt/response.

2. **Phased Workflow Core** (3-4 days):  
   Classes for each phase (e.g., ResearchPhase.new(query).execute → JSON).  
   Mock RAG pulls from rag-structure YAML/JSON (no live fetchers needed for spike).

3. **RAG Validation** (1 day):  
   Test phase queries against mocked provider; confirm context relevance (e.g., vision ties in plan phase). Flag insufficiencies (e.g., if semi-static empty, note for automation epic).

4. **Testing & Metrics** (2 days):  
   Run on samples (e.g., "Extend holdings sync for options"); score PRDs manually. Compare to baseline single-shot.

5. **Documentation & Review** (1 day):  
   Log phase outputs; human review all artifacts. Document findings (e.g., "Agentic reduced hallucinations by 50%; RAG sufficient but deltas needed").

**Dependencies**: Existing rag-structure (use as-is); Ollama setup on M3 Ultra.

**Risks & Mitigations**:
- Ollama reasoning weak: Fallback to Claude 4.5 for spike runs; note in report.
- RAG gaps: If deltas missing cause issues, defer to post-spike epic (e.g., git-hook summarizer).
- Latency: Off-hours runs; limit phases if >20s.

## Benefits & Trade-offs
- **Benefits**: Zero-cost validation; better PRD quality via phases; RAG byproduct insights for automation.
- **Trade-offs**: Added latency (multi-calls); potential underperformance on complex reasoning—address post-spike.
