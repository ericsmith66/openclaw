# Epic REFACTOR-001: Codebase Architectural Refactoring

This epic defines a systematic refactoring effort to address technical debt, improve maintainability, and enforce SOLID principles across the NextGen Plaid codebase.

The refactoring focuses on **service extraction**, **DRY principles**, and **separation of concerns** while maintaining **backward compatibility** and ensuring all existing tests pass.

---

## Epic goal

Reduce complexity and improve maintainability by breaking down large service objects, extracting shared behavior, and establishing clear boundaries between business logic and infrastructure layers.

Target outcomes:
- Services under 300 lines
- No duplicated guardrail/validation logic
- Clear separation of concerns in channels and controllers
- Easier to test and reason about code

---

## Current state assessment

### Critical issues (High Priority)
1. **SapAgent (927 lines)**: God object with 15+ public methods mixing concerns
2. **AiWorkflowService (825 lines)**: Multiple responsibilities (orchestration, agent building, context management, artifact writing)
3. **AgentHubChannel (599 lines)**: Business logic embedded in channel (should be service layer)
4. **SmartProxyApp (962 lines)**: Fat Sinatra controller mixing HTTP routing, tool orchestration, response transformation

### Important improvements (Medium Priority)
5. **Tool classes**: Duplicated `enforce_tool_guardrails!` across SafeShellTool, GitTool, etc.
6. **SdlcValidationScoring (412 lines)**: Handles gathering, validation, scoring, and reporting
7. **CSV Importers**: Three similar importers with repeated patterns (354, 234, 165 lines)

---

## Guiding principles

1. **Backward compatibility**: All existing tests must pass after each PRD
2. **Incremental refactoring**: Small, atomic changes that can be reviewed independently
3. **Extract before delete**: Create new classes/modules before removing old code
4. **Test coverage**: Maintain or improve test coverage through refactoring
5. **No feature changes**: Pure refactoring; no new functionality
6. **Each refactor should stand on its own**: commit each PRD to a separate branch off of master
7. **Verify sufficent test coverage before attempting the refactor**

---

## Refactoring strategy

### Phase 1: Service extraction (High Priority)
- Extract distinct responsibilities from large service objects
- Create clear public interfaces for new services
- Maintain backward-compatible facades in original files

### Phase 2: DRY improvements (Medium Priority)
- Extract common patterns to mixins/base classes
- Centralize configuration and constants
- Remove code duplication

### Phase 3: Architecture improvements (Low Priority)
- Formalize state machines
- Improve logging infrastructure
- Centralize configuration management

---

## Dependencies

- All existing functionality remains unchanged
- All existing tests must pass
- No changes to public APIs or controller endpoints
- Database schema remains unchanged

---

## Risk mitigation

### Technical risks
- **Risk**: Breaking existing functionality during refactoring
  - **Mitigation**: Comprehensive test suite must pass after each PRD; feature flags for gradual rollout

- **Risk**: Performance regression from additional layers
  - **Mitigation**: Benchmark critical paths before/after; maintain object allocation budgets

- **Risk**: Merge conflicts during active development
  - **Mitigation**: Coordinate with team; refactor less-frequently-modified files first

### Process risks
- **Risk**: Scope creep (adding features during refactoring)
  - **Mitigation**: Strict "refactor only" rule; separate tickets for feature work

## Implementation order (PRD sequence)

The refactoring is broken into atomic PRDs in priority order:

1. **PRD-REFACTOR-001A**: Extract SapAgent responsibilities
   - Extract 5 service classes from SapAgent
   - Target: reduce SapAgent to < 200 lines (facade/coordinator)

2. **PRD-REFACTOR-001B**: Refactor AiWorkflowService
   - Extract AgentFactory, ContextManager, GuardrailEnforcer
   - Move ArtifactWriter to separate file
   - Target: reduce AiWorkflowService to < 300 lines

3. **PRD-REFACTOR-001C**: Extract AgentHubChannel business logic
   - Extract ChatHandler, CommandExecutor, AutonomousCommandService
   - Target: reduce AgentHubChannel to < 200 lines

4. **PRD-REFACTOR-001D**: DRY up tool guardrails
   - Create Agents::ToolGuardrailsMixin
   - Apply to SafeShellTool, GitTool, CodeAnalysisTool, ProjectSearchTool
   - Target: eliminate ~30 lines of duplication per tool

5. **PRD-REFACTOR-001E**: Extract SdlcValidationScoring responsibilities
   - Extract EvidenceGatherer, Validator, Scorer, ReportGenerator
   - Target: reduce to < 100 lines per class

6. **PRD-REFACTOR-001F**: Consolidate CSV importers
   - Create BaseCsvImporter abstract class
   - Extract common patterns (validation, error handling, transactions)
   - Target: reduce each importer to < 100 lines

7. **PRD-REFACTOR-001G**: Extract SmartProxy business logic
   - Extract ResponseTransformer, ToolOrchestrator, ToolExecutor, ModelRouter, ModelAggregator, RequestAuthenticator
   - Target: reduce app.rb to < 350 lines
   - Benefit: Enables clean implementation of PRD-AH-OLLAMA-TOOL-01
