# Epic AH-012: Agent SDLC Testing CLI

This epic defines a CLI-driven, end-to-end autonomous testing harness for the SAP → Coordinator → CWA SDLC workflow.

The CLI mirrors existing UI-driven flows, but runs in **autonomous test mode** (no human approvals; auto-approve on success) while persisting run evidence to **DB + filesystem** for later inspection and prompt/RAG/tool honing.

---

## Epic goal

Provide a repeatable way to execute the full SDLC workflow from the command line with **controlled variations** (inputs/prompts/models/RAG/tools), **durable evidence** (Artifact + AiWorkflowRun + filesystem logs), and **objective validation/scoring**.

---

## Dependencies

- AH-009 (Agent-Hub-Workflow-MVP)
- Existing tables: `artifacts`, `ai_workflow_runs`
- Local Ollama via SmartProxy
- Existing orchestration: `AiWorkflowService` and `Agents::Registry` (no interface breaking changes)

---

## Ground truth for transitions (autonomous)

- Canonical success-path transition logic: `AiWorkflowService#finalize_hybrid_handoff!`
  - On success paths, the service advances state via `artifact.transition_to("approve", <persona>)`.
- Low-level executor and validator: `Artifact#transition_to`
- UI gating in production: typically human-driven via Agent Hub confirmation actions.
- CLI runs: **bypass human gates**.
  - Auto-approve on success.
  - No `awaiting_review` / human-confirm states in the happy path.
  - Escalations happen only on explicit failure conditions.

In `test_mode` (CLI runs), “bypass human gates” means:

- Skip ActionCable broadcasts (no UI updates).
- Force auto-approve on success in `AiWorkflowService#finalize_hybrid_handoff!`.
- Bypass environment toggles that normally disable tool execution (e.g., treat tool execution as enabled for the run).
- Suppress “escalate to human” failures (log them as evidence, then continue with default/retry behavior).

---

## Definition of “autonomous success” per phase

Core definition (required for advancement):

- No unhandled exceptions raised (including `GuardrailError` / `EscalateToHumanError`)
- Phase advanced via `transition_to("approve", <persona>)` on the success path

Additive validation signals (recorded; not required for advancement until PRD-012E):

- `backlog` / `ready_for_analysis`: PRD generated and `payload["content"]` present (SAP strategy validation)
- `in_analysis`: `payload["micro_tasks"]` is a non-empty array with minimal schema `{id, title, estimate}`
- `in_development`: green tool outcomes (e.g., tests pass via `SafeShellTool`) + `payload["implementation_notes"]` populated
- `ready_for_qa` / `complete`: rubric score `>= 7` (PRD-012E) + no pending escalations

## Persistence model for test runs

- **DB**: `Artifact` + `AiWorkflowRun` records persist for successful runs.
- **Rollback**: wrap run execution in a transaction; rollback DB changes on any exception.
- **Filesystem**: logs + artifact snapshots always persist as durable test evidence.

Primary human-readable output for a run is Markdown:

- `knowledge_base/test_artifacts/<run_id>/run_summary.md`

Machine-readable validation/scoring details are persisted separately (see PRD-012E).

---

## Run identity

A single `run_id` is used across DB + filesystem:

- Format: `YYYYMMDD-HHMMSS-<secure_random_hex(6)>` (example: `20260112-111345-abc123`)
- Used for:
  - `AiWorkflowRun.correlation_id` (canonical)
  - Artifact payload metadata (e.g., `payload["test_correlation_id"]`)
  - Filesystem directories:
    - `knowledge_base/logs/cli_tests/<run_id>/...`
    - `knowledge_base/test_artifacts/<run_id>/...`

---

## Variations (scope)

Single-run overrides only:

- LLM configuration per persona/phase (model/prompt/RAG tiers)
- Sandbox level (`strict`/`loose`)
- Tool guardrails (e.g., `max_tool_calls` override)

No built-in matrix runner; users can script multiple invocations externally.

---

## Stage isolation (`--stage`)

`--stage=<artifact_phase>` is **phase-based only** (it always refers to an Artifact phase, e.g. `in_analysis`).

`--stage=<artifact_phase>` runs from a clean start:

1. Create a fresh Artifact in `backlog`.
2. Fast-forward to the target phase via synthetic `transition_to("approve", <persona>)` calls.
3. Inject minimal synthetic payload needed to enter the stage (e.g., fake `payload["content"]` for analysis-only).

Initial version does **not** support resuming an existing run.

---

## Run directory contract (filesystem outputs)

All durable run evidence lives under:

- `knowledge_base/test_artifacts/<run_id>/`

Minimum expected structure:

- `run_summary.md` (primary human-readable report)
- `prd.md`
- `micro_tasks.json`
- `plan_summary.md`
- `cwa_summary.md`
- `code_diffs/`
- `handoffs/`
- `validation.json`
- `logs/` (optional convenience copy; canonical logs live under `knowledge_base/logs/cli_tests/<run_id>/`)

Canonical structured JSON logs live under:

- `knowledge_base/logs/cli_tests/<run_id>/` (e.g., `cli.log`, `sap.log`, `coordinator.log`, `cwa.log`, `summary.log`)

---

## RAG tiers (v1 mapping + truncation)

Tier-to-path mapping (v1):

- `foundation` → `knowledge_base/static_docs/`
- `structure` → `knowledge_base/schemas/`
- `history` → `knowledge_base/inventory.json` + recent run summaries/logs (7-day lookback)

Concatenate tiers in order.

Truncation: v1 uses a deterministic **character cap** proxy (e.g., 100k chars) and logs when truncating.
