## Epic Goal

Make Legion able to **run a complete PRD implementation cycle automatically** — decompose, architect-review, code, score, retry — without human intervention between steps, while enforcing quality thresholds through automated gates.

At the end of Epic 2, you can point `bin/legion implement` at a PRD path and walk away. A **Conductor Agent** — an LLM-powered orchestrator — reads the current workflow state and decides the next phase by calling orchestration tools. It decomposes the PRD into tasks, has the Architect review the plan, dispatches tasks in parallel via Solid Queue, runs QA scoring gates, auto-retries failures with accumulated context, and produces a scored, auditable artifact chain. Every decision the Conductor makes — and *why* it made it — is persisted to the database. The human intervenes only when the Conductor escalates.

**This is the WorkflowEngine, and it's a prompt — not code.** Epic 1 built the dispatch pipeline, event persistence, and dependency graph. Epic 2 adds a Conductor Agent that orchestrates the full loop (Decompose → Architect Review → Code → QA → Retry) by following prompt-defined rules and calling tool-guarded actions. The workflow logic lives in an editable Liquid prompt template, not in Ruby state machine code. Changing the workflow means editing a prompt, not writing a migration.

1. `bin/legion execute-plan` dispatches all ready tasks concurrently via Solid Queue (Tasks 1+2 run simultaneously, then 3+4, etc.)
2. `bin/legion score --workflow-run <id>` dispatches QA agent and returns a numeric score (0-100) with structured feedback, stored as real Artifact
3. Failed tasks can be reset to `pending` and re-dispatched with accumulated error context

7. PromptBuilder produces phase-specific prompts from Liquid templates with strict variable validation
8. QualityGate base class parses LLM score output and enforces configurable thresholds
9. ArchitectGate blocks plans scoring < 90 (between decompose and code); QAGate blocks code scoring < 90
10. Failed gates trigger auto-retry: two distinct limits (execution max 3 cycles, task-level configurable)
11. `bin/legion implement <prd-path>` runs full loop including retrospective
12. Per-project advisory lock prevents concurrent workflows (PostgreSQL advisory lock)
13. Conductor prompt template (`conductor_prompt.md.liquid`) is editable — workflow changes by editing prompt, not code
14. WorkflowEngine (outer) / ConductorService (inner) service layering with event-driven ConductorJob
15. ConductorHeartbeatJob detects stale tasks (15 min default) and re-dispatches Conductor
16. Parallel file conflict is a blocking validation (auto-serializes conflicting tasks)
17. `--dry-run` outputs PRD hash, rendered prompt, expected tool call, lock status, concurrency mode
18. Every `implement` run produces a `retrospective_report` Artifact with recommendations across 6 categories
19. Three-layer testing strategy: tool guard unit tests, prompt contract tests, VCR integration tests
20. All tests pass, RuboCop clean

## Conductor Agent Architecture

### The Core Idea: The Workflow Engine Is a Prompt

Traditional workflow engines encode transitions in code. Legion takes a different approach: **the WorkflowEngine is a Conductor Agent** — an LLM that reads the current workflow state and decides what to do next by calling orchestration tools.

The workflow logic lives in `app/prompts/conductor_prompt.md.liquid` (D-26), not in Ruby. Changing the workflow means editing a prompt template, not writing code + tests + migration.

### Service Layering (D-27)

| Service | Responsibility | Called By |
|---------|---------------|-----------|
| **WorkflowEngine** | Public outer service. Owns the `implement` lifecycle: create WorkflowExecution, acquire advisory lock, capture PRD snapshot, enqueue first ConductorJob, provide status API | `bin/legion implement`, CLI status commands |
| **ConductorService** | Inner service. Single Conductor dispatch: assemble Conductor agent from `conductor.yml`, dispatch with current execution state, process tool call, create ConductorDecision | `ConductorJob` (Solid Queue) |

### Two-Layer Architecture

| Layer | What Decides | How | Determinism |
|-------|-------------|-----|-------------|
| **Conductor Agent** (orchestration) | Which phase comes next, whether to retry or escalate | LLM follows prompt rules, calls orchestration tools | Guided by prompt rules (high predictability), but can apply judgment |
| **Worker Agents** (execution) | How to write code, how to score, how to review | LLM uses coding/QA tools via agent_desk Runner | Non-deterministic (creative work) |

### Event-Driven Dispatch (D-20)

The Conductor runs as **ConductorJob** in Solid Queue — not synchronously. Each trigger enqueues a new ConductorJob:

```
bin/legion implement PRD-2-01.md
  │
  ├── WorkflowEngine creates WorkflowExecution
  ├── WorkflowEngine enqueues ConductorJob(trigger: :start)
  ├── CLI enters polling loop (WorkflowExecution.status every 2s)
  │
  │   [BACKGROUND — Solid Queue]
  │   ConductorJob → Conductor calls dispatch_decompose
  │     → DecompositionService completion callback → enqueues ConductorJob(trigger: :decomposition_complete)
  │
  │   ConductorJob → Conductor calls dispatch_architect_review
  │     → ArchitectGate completion callback → enqueues ConductorJob(trigger: :architect_review_complete)
  │
  │   ConductorJob → Conductor calls dispatch_coding
  │     → TaskDispatchJob completions → all terminal? → enqueues ConductorJob(trigger: :all_tasks_complete)
  │
  │   ConductorJob → Conductor calls dispatch_scoring
  │     → QA scoring callback → enqueues ConductorJob(trigger: :scoring_complete)
  │
  │   ... (retry/retrospective/complete)
  │
  ├── CLI sees status=completed → print summary → exit 0
  └── CLI killed? Workflow continues in background. Reconnect: bin/legion status --execution <id>
```

### Heartbeat / Stale Task Detection (D-18)

**ConductorHeartbeatJob** — Solid Queue recurring job (every 60 seconds):
- Finds all WorkflowExecutions with `status: :running`
- For each, checks tasks with `status: :running` and `started_at` older than timeout
- Default timeout: 15 minutes (configurable via `metadata["heartbeat_timeout_minutes"]`)
- Stale tasks reset to `pending`, `retry_count` incremented, `last_error` set to "Stale task — heartbeat timeout"
- Enqueues ConductorJob to re-evaluate

### Conductor Agent Identity (D-17)

The Conductor is a **dedicated agent config** — `.aider-desk/agents/conductor.yml` — with:
- Model: Claude Sonnet (cost-efficient for routing decisions)
- Tool approvals: orchestration tools only (no coding tools)
- Rules: conductor-specific orchestration rules
- Imported via `TeamImportService` alongside other agents
- `ConductorDecision.input_summary` includes `"model"` key to record which model decided

### Conductor Orchestration Tools (D-6, D-19, D-29)

The Conductor has access to these tools — and **only** these tools. ALL tools accept a required `reasoning:` string parameter (D-19). Each tool validates preconditions before executing:

| Tool | What It Does | Precondition Guard |
|------|-------------|-------------------|
| `dispatch_decompose(reasoning:)` | Dispatches Architect agent to decompose PRD into tasks | Execution in `decomposing` phase, no tasks yet (or re-decompose after architect feedback) |
| `dispatch_architect_review(reasoning:)` | Dispatches Architect to review the decomposition plan (D-29) | Execution in `architect_reviewing` phase, tasks exist, decomposition_attempt < 2 |
| `dispatch_coding(reasoning:)` | Dispatches all ready tasks. Parallel (Solid Queue) or sequential per flags. | Execution in `coding` or `retrying` phase, ready tasks exist |
| `dispatch_scoring(reasoning:)` | Dispatches QA agent to score the implementation | Execution in `scoring` phase, all tasks in terminal state |
| `retry_with_context(reasoning:)` | Accumulates gate feedback, resets targeted tasks, increments attempt | Score below threshold, attempt < max_retries |
| `run_retrospective(reasoning:)` | Triggers end-of-run analysis, produces `retrospective_report` Artifact | Execution in `retrospective` phase |
| `mark_completed(reasoning:)` | Marks execution as completed with final score | Score ≥ threshold, retrospective complete |
| `escalate(reasoning:)` | Marks execution as escalated | Called when max retries/decompositions exceeded or unrecoverable error |
| `get_execution_status(reasoning:)` | Returns current execution state (read-only) | Always available |

## Parallel Execution via Solid Queue

### Job: `TaskDispatchJob`
- Receives: `task_id`, `workflow_execution_id`
- On perform: marks task `running`, sets `started_at`, calls DispatchService, updates status
- On success: marks task `completed`, sets `completed_at`, checks if more tasks ready → enqueue them, checks if ALL terminal → enqueue ConductorJob
- On failure: marks task `failed`, stores error in `last_error`, checks if all terminal → enqueue ConductorJob

### File Conflict Validation (D-25)
Before enqueuing a parallel wave, check for overlapping file paths between ready tasks. Conflicting tasks are automatically serialized (second held in `pending` until first completes). Logged: "Tasks #3 and #5 both reference `app/models/user.rb` — serializing."

### Worker Configuration

```yaml
# config/solid_queue.yml
production:
  workers:
    - queues: [task_dispatch, conductor]
      threads: 3
      processes: 1
  dispatchers:
    - polling_interval: 1
  recurring:
    conductor_heartbeat:
      class: ConductorHeartbeatJob
      schedule: every 1 minute
```

```
# Procfile.dev (D-20)
web: bin/rails server
worker: bundle exec rake solid_queue:start
```

## Retry Logic (D-30)

### Two Distinct Retry Scopes

**Execution-level (outer loop):**
- `WorkflowExecution.attempt` — counts complete CODE → QA → RETRY cycles
- Max 3 (default, configurable via `--max-retries`)
- After 3 full cycles: retrospective → escalate

**Task-level (inner granularity):**
- `Task.retry_count` — counts per-task reset-to-pending events
- Max configurable per execution (`WorkflowExecution.task_retry_limit`, default 3)
- Task exceeding limit: marked `failed` permanently
- Conductor decides: can execution proceed without this task, or escalate?

### Context Accumulation

Each retry gets richer context: original prompt + all prior QA feedback + error history.

## Testing Strategy (D-31)

### Layer 1 — Tool Guard Unit Tests (deterministic, no LLM)
- Each orchestration tool's precondition guards
- `mark_completed` refuses score < threshold
- `dispatch_coding` refuses if no ready tasks
- Fast, run on every CI build

### Layer 2 — Prompt Contract Tests (live LLM, no VCR)
- ~5-8 scenario tests verifying prompt behavioral contracts
- "Given score=87, threshold=90, attempt=1 → Conductor calls retry_with_context"
- Run when prompt changes, not every CI build
- Tagged `live_llm: true`

### Layer 3 — Integration Tests (VCR)
- Full cycle tests recording service-layer behavior
- Assert state changes (phases transitioned, Artifacts created, ConductorDecisions persisted)
- Do NOT assert on ConductorDecision.reasoning text (LLM output varies)

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Solid Queue worker crashes mid-task | High — partial file edits | ConductorHeartbeatJob detects stale tasks (15 min timeout, D-18). Reset to pending for re-dispatch. |
| QA agent score output unparseable | High — gate can't function | Strict Liquid prompt template enforces format. Multi-pattern parser with fallback. Score 0 on parse failure. |
| Parallel agents create file conflicts | High — merge conflicts | Blocking file conflict validation (D-25). Conflicting tasks auto-serialized. |
| Advisory lock starvation | Medium — blocks other PRDs | Lock timeout (default 4h). Stale lock detection. Manual override: `bin/legion unlock`. |
| Conductor makes wrong phase decision | Medium — invalid workflow state | Tool precondition guards prevent invalid transitions. ConductorDecision trail for debugging. |
| Conductor callback fails to fire | Medium — workflow stuck | ConductorHeartbeatJob (60s) detects stale executions and re-triggers Conductor. |
| Retry context too long | Medium — context window pressure | Cap at 2000 tokens per retry. Summarize prior feedback. |
| Liquid template missing variable | Low — prompt rendering fails | Liquid strict mode raises immediately with missing variable name. |
| Conductor adds latency per transition | Low — 2-5s per decision | Short-lived, one call per transition. Acceptable given phases take minutes. |
| Prompt drift breaks workflow | Medium — workflow stops | Templates version-controlled. Prompt contract tests (Layer 2, D-31) verify behavior. |
| Gate threshold too high (90) | Low — excessive retries | Configurable per execution via `--threshold`. |
| PRD modified mid-execution | Low — inconsistent retries | PRD snapshot + content hash (D-23). Warn on drift. |
| Retrospective context too large | Medium — expensive LLM call | Documented as known concern. Monitor during implementation. Add caps if needed (S-5 deferred). |
