## Epic Goal

Make Legion able to **dispatch any agent to any task, record everything that happens, and decompose complex work into atomic tasks with a dependency graph** — all from the command line.

At the end of Epic 1, you can point the Architect at a PRD and get back an ordered, scored, dependency-aware task list. Then execute that plan — each agent running with its full identity (rules, skills, tool approvals, custom instructions, model) — with every event persisted to PostgreSQL and safety rails firing automatically.

**This is not the WorkflowEngine.** There is no automatic chaining (Plan -> Approve -> Code -> QA). The human is the orchestrator, deciding when to run the next command. Epic 2 automates the sequencing. Epic 1 builds everything Epic 2 needs underneath.

### What the Benchmarks Proved

20 live agentic coding tests (4 prompts × 5 models) through the gem's Runner + PowerTools pipeline — **20/20 success rate**. Key findings:

- **The gem works as-is** — no code changes needed for real coding tasks
- **Task sizing is the variable, not model choice** — all models pass well-scoped tasks
- **Reasoning models hurt agentic loops** — DeepSeek Reasoner was 55% slower than Chat
- **The atomic task scale** (files × concepts × dependencies, decompose if score > 6) predicts success
- **Three failure walls** — iteration cap, context window, time budget — each model hits a different one first
- **The MessageBus + Hooks + ApprovalManager are the real-time control plane** — the orchestrator watches and intervenes through these three channels

### The Gem's Existing Capabilities (No Changes Needed)

| Capability | Gem Component | How It Works |
|------------|---------------|--------------|
| Agent execution loop | `Runner.run()` | Sends prompt -> processes tool calls -> iterates until done |
| Skills (lazy, LLM-driven) | `SkillLoader.activate_skill_tool()` | Single tool `skills---activate_skill` — LLM chooses which skill to activate, full SKILL.md injected as tool result |

| Tool approval gates | `ApprovalManager` | ALWAYS / ASK / NEVER per tool — ASK blocks runner until authorized |
| Context management | `CompactStrategy`, `HandoffStrategy`, `TieredStrategy` | Auto-compacts or splits task when context grows too large |
| Agent self-tracking | `TodoTools` | In-memory scratch pad — LLM creates/checks off items during a single run, not persisted |

### Task Dependency Graph Example

Decomposing P4 (Polymorphic Soft Delete) — **test-first ordering**:

```text
Task 1:  "Write tests for Post soft-delete columns"              ─── independent (test)
Task 2:  "Write tests for Comment cascade-delete columns"         ─── independent (test)
Task 3:  "Add deleted_at to Post (make Task 1 tests pass)"       ← depends on [1] (code)
Task 4:  "Add deleted_at + cascade to Comment (make Task 2 pass)"← depends on [2] (code)
         ↓ (both must complete)                                          ↓
Task 5:  "Write tests for default_scope + with_deleted"           ← depends on [3, 4] (test)
Task 6:  "Implement default_scope + with_deleted (make pass)"     ← depends on [5] (code)
         ↓
Task 7:  "Write tests for soft_delete + restore"                  ← depends on [6] (test)
Task 8:  "Implement Post#soft_delete with cascade (make pass)"    ← depends on [7] (code)
Task 9:  "Implement Post#restore (make pass)"                     ← depends on [7] (code)
         ↓ (both must complete)                                          ↓
Task 10: "Write + run edge case tests"                            ← depends on [8, 9] (test)
```

Tasks 1 and 2 have no dependencies — **parallel-eligible** (both are test tasks).
Tasks 3 and 4 are parallel-eligible after their respective test tasks complete.
Tasks 8 and 9 both depend only on 7 — **parallel-eligible** after test task completes.
Each implementation task has tests already written — the agent runs them (red), codes until green.

In Epic 1, `execute-plan` dispatches one ready task at a time (sequential).
In Epic 2, Solid Queue dispatches all ready tasks concurrently — Tasks 1+2 run simultaneously, then 3+4, then 5, then 6, then 7, then 8+9, then 10. **Same data model, different dispatcher.**

## CLI Commands

### `bin/legion execute`

Dispatches a single agent to a single task.

```bash
bin/legion execute --team ROR --agent rails-lead --prompt "Add a User model with email validation"
bin/legion execute --team ROR --agent qa --prompt "Score this implementation..."
bin/legion execute --team ROR --agent architect --prompt "Review this plan..."
bin/legion execute --team ROR --agent debug --prompt "Fix these test failures..."
bin/legion execute --team ROR --agent rails-lead --prompt-file path/to/prompt.md
```

**Assembly pipeline:**
1. Load TeamMembership from DB → `to_profile` → Profile
2. `RulesLoader.load_rules_content(profile_dir_name:, project_dir:)` → XML rules
3. `PromptsManager.system_prompt(profile:, project_dir:, rules_content:, custom_instructions:)` → system prompt
4. `PowerTools.create(project_dir:, profile:)` + `SkillLoader.activate_skill_tool(project_dir:)` + `TodoTools.create` + `MemoryTools.create` → combined ToolSet
5. `ModelManager.new(provider:, model:, api_key:, base_url:)` → SmartProxy connection
6. Create WorkflowRun (status: `running`)
7. `PostgresBus.new(workflow_run:)` → event persistence
8. `HookManager.new` + orchestrator hooks → safety rails
9. `ApprovalManager.new(tool_approvals:)` → tool gates (auto-approve or CLI-interactive)
10. `Runner.new(model_manager:, message_bus:, hook_manager:, approval_manager:, ...).run(...)`
11. Update WorkflowRun (status, duration, iterations, result)

### `bin/legion decompose`

Decomposes a PRD into scored, dependency-aware tasks.

```bash
bin/legion decompose --team ROR --prd knowledge_base/epics/wip/epic-1/.../PRD-1-01.md
bin/legion decompose --team ROR --prd path/to/any-prd.md --dry-run
```

**Process:**
1. Read PRD file
2. Dispatch Architect agent with decomposition prompt
3. Parse structured output (ordered tasks, agents, dependencies, scores)
4. Validate: flag any task with total_score > 6 for further decomposition
5. Create Task records + TaskDependency edges in DB
6. Print task list with dependency graph to console

**The decomposition prompt asks the Architect to:**
- Break the PRD into atomic coding tasks
- **Test-first ordering:** For each feature, produce the test task before the implementation task. The implementation task depends on the test task. Tests define the contract; the coding agent runs tests (red), writes code to make them pass (green).
- For each task: prompt text, recommended agent, files touched, concepts, cross-model dependencies
- For each task: list dependencies on other tasks in this list (by number)
- Mark tasks with no dependencies as `independent` (parallel-eligible)
- Two tasks that edit different files with no shared model dependencies CAN run in parallel

### `bin/legion execute-plan`

Executes a decomposed task list respecting the dependency graph.

```bash
bin/legion execute-plan --workflow-run &lt;id&gt;
bin/legion execute-plan --workflow-run &lt;id&gt; --start-from &lt;task-id&gt;
```

**Process:**
1. Load all Tasks for the WorkflowRun
2. Find ready tasks (all dependencies completed, status `pending`)
3. Dispatch first ready task via the full assembly pipeline (same as `bin/legion execute`)
4. On completion: mark task `completed`, update execution_run_id
5. On failure: mark task `failed`, halt (or `--continue-on-failure` flag)
6. Loop: find next ready task(s), dispatch
7. Complete when all tasks are `completed` or no more ready tasks exist

Epic 1: dispatches one ready task at a time.
Epic 2: dispatches all ready tasks concurrently via Solid Queue.

### `rake teams:import`

Imports agent configurations from filesystem to database.

```bash
rake teams:import[~/.aider-desk]
rake "teams:import[~/.aider-desk,--dry-run]"
```

## The Full Agent Assembly Pipeline

This is the critical path that makes each agent run with its complete identity.
The CLI dispatch, execute-plan, and decompose commands all use this same pipeline.

```
TeamMembership (from DB)
  |
  +-> to_profile -> AgentDesk::Agent::Profile
  |     (provider, model, max_iterations, use_* flags, tool_approvals,
  |      tool_settings, custom_instructions, subagent_config)
  |
  +-> RulesLoader.load_rules_content(profile_dir_name:, project_dir:)
  |     3-tier: global agent -> project-wide -> project agent
  |     Output: XML string for system prompt injection
  |
  +-> PromptsManager.system_prompt(profile:, project_dir:, rules_content:, custom_instructions:)
  |     Liquid template -> &lt;Knowledge&gt;&lt;Rules&gt;, tool permissions, workflow, custom instructions
  |
  +-> ToolSet assembly:
  |     PowerTools.create(project_dir:, profile:)
  |     + SkillLoader.activate_skill_tool(project_dir:)  <- LLM activates skills lazily
  |     + TodoTools.create                                <- LLM self-tracking scratch pad
  |     + MemoryTools.create (if use_memory_tools)
  |
  +-> ModelManager.new(provider:, model:, api_key:, base_url:)
  |     -> SmartProxy connection
  |
  +-> PostgresBus.new(workflow_run:)
  |     -> Persists all events to WorkflowEvent table
  |     -> In-process callbacks for orchestrator
  |     -> Solid Cable broadcast (noop until Epic 4)
  |
  +-> HookManager + orchestrator hooks
  |     -> Iteration budget, context pressure, handoff capture, cost limits
  |
  +-> ApprovalManager.new(tool_approvals:)
  |     -> ALWAYS / ASK / NEVER per tool
  |     -> ASK blocks runner until CLI-interactive approval
  |
  +-> Runner.new(model_manager:, message_bus:, hook_manager:, approval_manager:, ...)
        .run(prompt:, system_prompt:, tool_set:, profile:, project_dir:,
             agent_id:, task_id:, max_iterations:)
```

### How Skills Work (Unchanged — Gem Handles It)

Skills are **not pre-loaded** into the system prompt. They work like this:

1. `SkillLoader.discover(project_dir:)` finds all SKILL.md files (global + project)
2. `activate_skill_tool` creates a single tool: `skills---activate_skill`
3. The tool's description lists all available skills with names and descriptions
4. The LLM decides when a skill is relevant and calls `skills---activate_skill` with `{"skill": "Rails Minitest + VCR"}`
5. The full SKILL.md content is injected into the conversation as a tool result
6. The LLM follows the skill instructions in its subsequent responses

The orchestrator doesn't need to know which skills to load. The profile's `use_skills_tools: true` flag gates whether the agent sees skills at all. The LLM handles selection.

### How Rules Work (Unchanged — Gem Handles It)

Rules ARE pre-loaded into the system prompt:

1. `RulesLoader.load_rules_content(profile_dir_name:, project_dir:)` reads .md files from the 3-tier hierarchy
2. Returns XML string: `&lt;File name="rails-base-rules.md"&gt;&lt;![CDATA[...]]&gt;&lt;/File&gt;`
3. `PromptsManager.system_prompt(...)` renders the Liquid template, injecting rules into `&lt;Knowledge&gt;&lt;Rules&gt;`
4. Every LLM call includes these rules in the system message

## Orchestrator Hooks (Safety Rails)

Registered on `HookManager` before each `runner.run()`:

| Hook | Fires When | Orchestrator Action |
|------|-----------|---------------------|
| `on_tool_called` | Before every tool execution | Count iterations. If approaching model's decomposition threshold (Grok: >100, Claude: >50, DeepSeek: >30), log warning on WorkflowRun. |
| `on_token_budget_warning` | Context window threshold crossed | At 60%: mark WorkflowRun `at_risk`. At 80%: block default compaction, mark `decomposing`. |
| `on_handoff_created` | Gem auto-splits task (HandoffStrategy) | Create new WorkflowRun for continuation. Mark original `handed_off`. |
| `on_cost_budget_exceeded` | Cost ceiling hit | Stop run. Status: `budget_exceeded`. |

These are rules-based guardrails — no LLM judgment, just threshold checks.
Epic 2 adds intelligent responses (auto-decompose, model switching).

## The Atomic Task Scale (Built Into Decomposition)

### Three Dimensions (1-4 each)

| Dimension | 1 (Atomic) | 2 (Coupled) | 3 (Compound) | 4 (Orchestration) |
|-----------|-----------|-------------|---------------|-------------------|
| **Files Touched** | 1-2 | 3-4 | 5-7 | 8+ |
| **Concept Count** | 1 | 2 | 3-4 | 5+ |
| **Cross-Model Deps** | 0 | 1 | 2-3 | 4+ |

### The Rule

> **Total score > 6 → decompose further.**

The decomposition prompt instructs the Architect to score each task.
The parser validates scores and flags any task above threshold.

### Model-Specific Iteration Thresholds (From Benchmark Data)

| Model | Decompose If > | Hard Limit (iters) | Limiting Wall |
|-------|---------------|---------------------|---------------|
| Grok 4.1 Fast | 100 iterations | 200 (cap) | Iteration cap |
| Claude Sonnet 4.6 | 50 iterations | ~98 | Context window |
| Qwen3 Coder Next | 55 iterations | ~110 | Time budget |
| DeepSeek Chat/Reasoner | 30 iterations | ~62 | Context window |

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Architect decomposition output is inconsistent/unparseable | High | Define strict output format in prompt. Validate aggressively. Fallback to manual task creation. |
| TeamMembership JSONB → Profile conversion misses fields | High | Comprehensive mapping tests against filesystem ProfileManager output. |
| DAG cycle detection performance on large task graphs | Low | Tasks per PRD: 5-20. Not a scaling concern in Epic 1. |
| PostgresBus write throughput during streaming (many response.chunk events) | Medium | Batch inserts or skip response.chunk persistence (configurable). |
| Full assembly pipeline too slow (too many components to wire) | Medium | Profile once, extract into service object if needed. |
| Decomposition prompt quality varies by model | Medium | Use Architect (Claude Opus) consistently for decomposition — benchmark showed best reasoning per turn. |

## Design Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| **D-1** | JSONB config on TeamMembership, not normalized columns | Agent configs have 20+ fields that vary. JSONB avoids schema churn. `to_profile` handles conversion. |
| **D-2** | DAG via join table (TaskDependency), not single FK | Enables many-to-many dependencies. A task can depend on multiple predecessors. Multiple tasks can be ready simultaneously (parallel-eligible). |
| **D-3** | Sequential execution in Epic 1, parallel data model | Costs almost nothing to design correctly now. Massive productivity unlock when Epic 2 adds Solid Queue dispatch. |
| **D-4** | PostgresBus wraps CallbackBus (not replaces) | In-process callbacks still needed for orchestrator hooks during `runner.run()`. DB persistence is additive. |
| **D-5** | Skills are lazy (LLM-driven), rules are eager (system prompt) | Matches AiderDesk behavior. Skills as tool results avoid bloating system prompt. Rules are always-relevant context. |
| **D-6** | No Artifact model in Epic 1 | Results stored as WorkflowRun.result text. Structured artifact storage (with types, versioning, parent chains) belongs in Epic 2 where the WorkflowEngine produces typed outputs. |
| **D-7** | Architect agent does decomposition (not rules-based parser) | PRDs are natural language with inconsistent structure. The Architect understands code dependencies. Rules-based parsing is brittle. LLM + validation is more robust. |
| **D-8** | Atomic scale scoring stored on Task records | Enables future analysis: correlation between score and actual iterations/failures. Feeds the Iteration Budget Estimator in Epic 2. |
| **D-9** | Test-first task ordering in decomposition | The Architect produces test tasks before implementation tasks. Tests define the contract. The coding agent runs tests (red), writes code (green), iterates via tool feedback. This is what agentic coding is best at — the iterative self-correction loop. QA scoring shifts from "did you miss anything?" to "did you follow the plan?" Much tighter feedback loop. The decomposition prompt explicitly requires: "For each feature, produce the test task before the implementation task. The implementation task depends on the test task." |

## Relationship to Epic 2 (WorkflowEngine Core)

Epic 1 builds the **foundation**. Epic 2 builds the **automation**.

| Epic 1 Provides | Epic 2 Adds |
|----------------|-------------|
| Task model with dependency DAG | WorkflowEngine walks the graph automatically |
| Sequential `execute-plan` | Parallel dispatch via Solid Queue |
| Manual `decompose` command | Auto-decomposition when hooks detect failure walls |
| PostgresBus event persistence | WorkflowEngine subscribes to events, makes decisions |
| Orchestrator hooks (detect) | WorkflowEngine hooks (respond) |
| CLI dispatch of individual agents | Automatic agent chaining (Plan -> Approve -> Code -> QA) |
| WorkflowRun per agent dispatch | WorkflowRun per phase, linked to parent workflow |
| Task.status updated by CLI | Task.status driven by state machine transitions |

The dependency graph, event pipeline, and full assembly pipeline are the foundation Epic 2 builds on. No rework needed — just new consumers of the same data and infrastructure.
