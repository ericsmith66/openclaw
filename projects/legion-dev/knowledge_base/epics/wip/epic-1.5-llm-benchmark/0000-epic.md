## Epic Goal

Use Legion's orchestration pipeline to **build the `agent_desk` Ruby gem from scratch** in a clean `legion-test` repo, driven only by the Epic 4C PRD specifications from agent-forge. Run multiple LLM combinations — **cheapest first** — to determine the minimum model quality required for complex multi-PRD agentic coding.

This is not a feature epic. It is a **controlled experiment** that validates:
1. Legion's decomposer/executor pipeline on real-world, multi-file, multi-PRD work
2. Whether cheap/local models (Ollama) can handle foundational framework construction
3. Where exactly cheap models break down and what the escalation path looks like
4. The quality of Legion's decomposition as the deciding factor for cheap model success

**Hypothesis:** If the decomposer produces high-quality atomic tasks, cheap coding models can execute them. If decomposition is poor, even expensive models struggle. **The decomposer is the deciding factor.**

### Why the Decomposer Is the Deciding Factor

The benchmark data from Epic 0 showed:
- **All models pass well-scoped tasks** — 20/20 success rate across 5 models
- **Task sizing is the variable, not model choice**
- **Atomic task scale** (files × concepts × dependencies, decompose if > 6) predicts success

The decomposer (Architect agent) takes a PRD and produces the atomic task list with dependency graph. If it:
- Produces tasks that are truly atomic (score ≤ 6) → cheap coders succeed
- Misses dependencies or creates over-scoped tasks → even expensive coders fail
- Gets the test-first ordering right → coding agents have tight red/green feedback loops

**We use DeepSeek (remote) as Architect/decomposer.** Fallback: Grok-latest. The decomposer is the ONE role we don't go cheap on initially — it's the control variable.

## Execution Plan

### Per-Team Run Protocol

For each team configuration:

1. **Fresh `legion-test` clone** — clean Rails 8 scaffold with empty `gems/agent_desk/`
2. **Import team config** — `rake teams:import` with the team's agent configs
3. **For each PRD in dependency order:**
   a. `bin/legion decompose --team <team> --prd <prd-path>` — Architect decomposes
   b. Review decomposition output — log task count, scores, parallel groups
   c. `bin/legion execute-plan --workflow-run <id>` — execute the decomposed plan
   d. After plan completes: run `cd gems/agent_desk && bundle exec rake test`
   e. Log: PRD status, test count, assertions, failures, iterations, duration, tokens
   f. If tests fail: allow debug agent one pass, then log final state
4. **After all PRDs:** full gem test suite, RuboCop check, comparison against answer key

### PRD Execution Order (4C-Core)

```
Round 1: PRD-0010 (scaffold)     — easiest, proves pipeline works
Round 2: PRD-0005 (test harness) — depends on 0010
Round 3: PRD-0020 (tools)     ┐
Round 4: PRD-0030 (hooks)     ├─ can theoretically parallel, run serial for clean measurement
Round 5: PRD-0095 (bus)       ┘
Round 6: PRD-0091 (model mgr)   — first HARD PRD, likely failure point for cheap models
Round 7: PRD-0090 (runner)       — HARDEST PRD, integrates all prior work
Round 8: PRD-0092a (tokens)      — hard but contained
Round 9: PRD-0092b (compaction)  — hard, depends on everything
```

### Decision Gates

| After PRD | Decision |
|-----------|----------|
| 0010 + 0005 | If Team C fails on scaffolding, abort — model is fundamentally unsuitable |
| 0020 + 0030 + 0095 | If Team C fails on medium PRDs, escalate to Team B for remaining hard PRDs only |
| 0091 | **Key decision point.** SSE streaming is the first real integration challenge. If Team C fails, swap coder to DeepSeek-Chat and re-run 0091 only. |
| 0090 | If both Team C and B fail on runner, escalate to Team A. If Team A also fails, the issue is decomposition quality, not model quality. |
| 0092b | Full suite passing → Phase 1 success. Move to Phase 2 assessment. |

## Scoring Rubric

### Per-PRD Metrics (Automated)

| Metric | How | Weight |
|--------|-----|--------|
| Tests pass (binary) | `bundle exec rake test` exit code | Gate (must pass to proceed) |
| Test count | Minitest run count | 15% |
| Assertion count | Minitest assertion count | 15% |
| Zero failures/errors | `0 failures, 0 errors` in output | Gate |
| RuboCop clean | `rubocop` exit code | 5% |

### Per-PRD Metrics (Logged)

| Metric | How | Purpose |
|--------|-----|---------|
| Decomposition task count | From `bin/legion decompose` output | Measures decomposition granularity |
| Average task score | Mean of atomic scale scores | Measures decomposition quality |
| Tasks flagged (score > 6) | Count from decomposition | Measures over-scoping |
| Total iterations | Sum across all tasks in plan | Cost proxy |
| Total duration (wall clock) | Start to finish of `execute-plan` | Speed measurement |
| Debug cycles needed | Number of debug agent invocations | Measures first-pass quality |
| Tokens consumed | From SmartProxy/Ollama logs | Actual cost measurement |

### Per-Team Aggregate

| Metric | Formula |
|--------|---------|
| PRDs completed | Count of PRDs with passing tests (out of 9) |
| Cumulative test count | Sum of all tests across all PRDs |
| Cumulative assertions | Sum of all assertions |
| Total cost | Sum of token costs (cloud) or $0 (Ollama) |
| Total wall-clock time | End-to-end for all 9 PRDs |
| Escalation count | How many times we swapped to a more expensive model |

### Answer Key Comparison (Post-Hoc)

After each team run, compare against agent-forge's existing implementation:

| Comparison | Method |
|------------|--------|
| Test count delta | legion-test tests vs. agent-forge's 752 |
| Assertion count delta | legion-test assertions vs. agent-forge's 2022 |
| Architectural similarity | Manual spot-check: do module boundaries, class names, method signatures align? |
| Missing coverage | Are there acceptance criteria from the PRD that the generated tests don't cover? |

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Decomposer produces over-scoped tasks → cheap coders fail | High | DeepSeek-Chat is strong at structured decomposition. If task scores are consistently > 6, switch to Grok-latest. Review decomposition output before executing. |
| Ollama models can't handle long PRDs in context | Medium | PRDs are 3-7KB each. Qwen3-Coder-Next supports 128K context. Should be fine. |
| PRDs reference AiderDesk TypeScript files that don't exist | Medium | Add a note to the knowledge base: "AiderDesk source references are informational only. Implement the Ruby equivalent based on the acceptance criteria." |
| Qwen3:32b QA scoring is too shallow (not verifying, just approving) | Medium | Compare Qwen3:32b QA reports against Legion's Claude-Sonnet QA reports on the same code. If Qwen3 consistently misses issues, swap QA first. Note: QA and Debug share the same model — both `qwen3:32b`. |
| SSE streaming (PRD-0091) requires Faraday + specific HTTP patterns | High | This is the expected failure point for cheap models. Have Team B ready for immediate escalation on this PRD. |
| Test contamination between PRDs (earlier PRD's tests break after later PRD) | Medium | Run full `rake test` after each PRD, not just the new tests. Log regression counts. |
| SmartProxy routing to Ollama models | Low | SmartProxy already supports Ollama backend. Verify routing config before first run. |
