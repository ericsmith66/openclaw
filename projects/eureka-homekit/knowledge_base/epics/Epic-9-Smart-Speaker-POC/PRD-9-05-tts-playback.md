### Overview

This PRD implements the text-to-speech (TTS) output pipeline. The eureka-homekit server (via Ollama on M3 Ultra) generates spoken audio using a custom "Eureka" voice personality. The Pi receives the audio (streamed or downloaded) and plays it through the HiFiBerry Amp4 + SB Acoustics speaker driver. A fallback to gTTS (Google Text-to-Speech) is provided if the server TTS is unavailable.

The goal is a distinct, high-quality Eureka voice that users immediately recognize as their smart home assistant.

---

### Requirements

#### Functional

- Receive TTS audio from the eureka-homekit server (generated by Ollama TTS model on M3 Ultra).
- Support two delivery modes:
  1. **Streamed**: Chunked audio over HTTPS (progressive playback as chunks arrive).
  2. **File download**: Full WAV/MP3 file via HTTPS URL, then play.
- Play audio through HiFiBerry Amp4 via ALSA (`aplay` or PyAudio).
- Support the custom "Eureka" voice (cloned/trained via Ollama with XTTS, Bark, or Coqui TTS).
- Provide local fallback TTS via gTTS (Google) for when server is unreachable.
- Play system chimes (wake acknowledgment, error, processing) from local WAV files.
- Control playback volume via ALSA mixer (`amixer`).
- Handle interruption: if user says "Hey Eureka" during playback, stop TTS and enter listening mode.

#### Non-Functional

- TTS audio quality: 22050 Hz or 44100 Hz, 16-bit PCM minimum.
- Playback start latency: < 500ms from audio receipt to first audible output (streamed mode).
- End-to-end TTS latency (server generation + network + playback start): < 2 seconds for short responses.
- No audible clicks, pops, or gaps during streamed playback.
- Volume level: configurable 0-100%, default 70%.
- Must not block the wake word listener during playback (concurrent audio I/O).

**TTS Client Module (`tts_client.py`):**
- Request TTS from server: `POST /api/v1/speaker/tts` with `{"text": "response text", "voice": "eureka"}`.
- Support streamed response: read chunked audio, feed to player progressively.
- Support file response: download WAV/MP3, then play.
- Timeout: 5 seconds for first byte; 30 seconds total.
- On failure: return None (caller falls back to local TTS).

**Audio Player Module (`audio_player.py`):**
- Play WAV/PCM audio through HiFiBerry ALSA device.
- Support blocking and non-blocking playback.
- Support interruption: expose `stop()` method that halts playback immediately.
- Use a dedicated playback thread so wake word listener continues running.

**Fallback TTS Module (`fallback_tts.py`):**
- Generate audio locally using gTTS: `gTTS(text, lang='en').save('/tmp/response.mp3')`.
- Convert to WAV if needed (gTTS outputs MP3).
- Play via audio player.
- Used when server is unreachable or TTS generation fails.

**Volume Control Module (`volume_control.py`):**
- Get/set volume via ALSA mixer: `amixer -c N sset 'Digital' XX%`.
- Expose `get_volume() -> int`, `set_volume(level: int)`, `mute()`, `unmute()`.
- Persist volume setting across reboots (write to config file).

**Server-Side TTS (Ollama on M3 Ultra):**
- Ollama runs a TTS model (e.g., XTTS-v2, Bark, or Coqui TTS) configured with a custom Eureka voice profile.
- Voice cloning: provide a reference audio sample of the desired Eureka voice (warm, friendly, slightly formal). Ollama uses this for voice cloning.
- The Rails API endpoint (`/api/v1/speaker/tts`) receives text, calls Ollama TTS, and streams audio back.
- Audio format: WAV (PCM 16-bit, 22050 Hz or 44100 Hz) for lowest latency. MP3 as compact alternative.

### Error Scenarios & Fallbacks

- **Server unreachable** → Fall back to gTTS (lower quality but functional). Log warning.
- **TTS generation timeout (> 5s for first byte)** → Abort server request, fall back to gTTS. Log warning.
- **Audio format unsupported** → Convert via pydub or ffmpeg subprocess. Log warning.
- **ALSA device busy** → Queue playback request, retry in 100ms. If busy for > 2 seconds, log error and skip.
- **Interruption during playback** → Stop playback immediately, clear audio buffer, return to listening state.
- **Volume at 0 / muted** → Still process TTS (don't skip API call), but playback will be silent. Log info.
- **Corrupt audio data** → Validate WAV header before playback. If corrupt, log error, play error chime, skip.

### Architectural Context

This PRD is the audio output counterpart to PRD 9-03 (audio input). Together they form the voice I/O loop:
```
[Wake Word (9-03)] → [STT (9-03)] → [API (9-06)] → [TTS (9-05)] → [Playback (9-05)]
         ↑                                                                  │
         └──────────────────── listening resumes ◄──────────────────────────┘
```

The audio player must coexist with the wake word listener. Since the ReSpeaker has AEC (Acoustic Echo Cancellation), the mic can still listen for "Hey Eureka" while the speaker is playing audio. However, the XVF3800's AEC must be verified to work with the HiFiBerry's I2S output.

Server-side TTS generation is handled by Ollama on M3 Ultra. The custom Eureka voice is an Ollama configuration concern, not an edge concern. The Pi only fetches and plays audio.

The chime system is independent of the TTS pipeline — chimes are local WAV files played immediately without server round-trip.

### Rollout / Deployment Notes

- gTTS requires internet access (Google Translate API). For fully offline fallback, consider pyttsx3 (lower quality, truly offline).
- Custom Eureka voice setup on Ollama requires:
  1. A reference audio sample (30-60 seconds of target voice).
  2. XTTS-v2 or equivalent voice cloning model loaded in Ollama.
  3. Voice profile configured and tested before Pi integration.
- Document the exact ALSA device name and card number for the HiFiBerry in `speaker/src/config.py`.
- Test AEC behavior: play music through speaker while recording mic — verify mic audio has reduced speaker bleed.
